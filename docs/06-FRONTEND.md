# 06 — Frontend Spec

A SaaS dashboard, not a chatbot. Users come here to manage campaigns and watch results.

## Stack
- Next.js 14 (App Router, `src/` dir, TypeScript)
- Tailwind + shadcn/ui
- TanStack Query (the only data layer — no Redux/Zustand)
- Recharts (analytics)
- react-dropzone (CSV upload)
- next/font (Geist)

## Layout shell
- Sidebar nav (fixed left): Dashboard, Campaigns, Leads, Templates, Integrations, Settings.
- Header: workspace name + Gmail-connected indicator.
- Main pane: page content.

Mobile responsive is **not** required for MVP.

---

## Pages

### `/dashboard`
Top-of-funnel stats and trend charts.

**Cards (5):** Total Leads, Emails Sent, Open Rate, Reply Rate, Meetings Booked.

**Charts:**
- Line chart: emails sent per day (last 30 days).
- Bar chart: reply rate by campaign.

Data source: `GET /api/campaigns` (aggregated server-side).

### `/campaigns`
Table of campaigns.

| Column | Source |
|--------|--------|
| Name | `name` |
| Status | badge (`draft / active / paused / completed`) |
| Leads | `stats.leads_count` |
| Sent | `stats.emails_sent` |
| Opened % | `stats.open_rate` |
| Replied % | `stats.reply_rate` |
| Created | `created_at` |

Row actions: View · Pause/Resume · Archive.
"New Campaign" button → modal launches the multi-step creation flow.

### `/campaigns/[id]`
Top: campaign stat strip (same metrics as the dashboard cards, scoped to this campaign).

Below: leads table — status, last touched, latest email subject. Clicking a lead opens a slide-out panel with the full email thread (every email + every reply, oldest-first).

### `/settings`
- Gmail OAuth connect button → hits `GET /api/auth/gmail` and redirects.
- Slack webhook URL input.
- API key inputs (Tavily, Firecrawl, etc.) — stored encrypted server-side.

## Campaign creation flow

A three-step in-modal form using a small React state machine (no form library).

### Step 1 — Basics
- Campaign name (text)
- Your product/service name (text)
- Value proposition (textarea, ≤ 100 words; counter)
- Case study (textarea — "We helped X achieve Y")

### Step 2 — Lead upload
- Drag-and-drop CSV via react-dropzone.
- Preview table: first 5 rows.
- Column mapping UI: map CSV columns to `company_name, website, contact_name, email`.
- Validation: highlight rows with missing/invalid email.

### Step 3 — Review & launch
- Summary: N leads, estimated send time, daily Gmail cap.
- **Launch Campaign** → `POST /api/campaigns` then `PATCH /api/campaigns/{id}/status` to `active`.
- On success → redirect to `/campaigns/{id}`.

Progress indicator (step 1/2/3) at the top of the modal.

---

## Data Layer
- Every server interaction goes through TanStack Query.
- Query keys follow `["resource", id?, filters?]` — e.g. `["campaigns", { page: 1 }]`, `["campaign", id]`.
- Mutations invalidate the relevant queries; do not optimistically update unless explicitly designed.
- No global stores. If a piece of UI state survives a remount, it lives in URL params.

## Rules
- Functional components only.
- No `console.log` in committed code.
- No third-party form library — the creation modal uses a custom state machine.
- Components in `components/ui/` are generated by shadcn — don't hand-edit unless overriding a primitive.
